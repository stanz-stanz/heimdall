Framework Options for the Mobile Console

How far do you want to go from the current vanilla JS?

A

Svelte + Vite (Full Rewrite)

Rewrite the console as a proper Svelte app with Vite build tooling. Component architecture, reactive stores for WebSocket state, built-in transitions for animations. The fullstack-guy's svelte.md patterns slot in directly.

Pros

  • Smallest bundle — compiles to vanilla JS, no runtime
  • Built-in transitions (fly, fade, slide) replace manual CSS
  • tweened stores for smooth metric animations
  • Reactive WebSocket store pattern from references
  • Component isolation — each card/widget is its own .svelte file

Cons

  • Requires build step (Vite dev server + build output)
  • Need to decide how to serve: static build output vs. Vite proxy to FastAPI
  • Full rewrite of ~1,700 lines
  • Smaller ecosystem than React (fewer off-the-shelf components)
B

Svelte Components, Keep the Shell

Keep the existing HTML shell and CSS but extract interactive pieces (queue cards, demo timeline, finding cards) into Svelte components mounted into the page. Incremental adoption — migrate one widget at a time.

Pros

  • Migrate gradually — no big-bang rewrite
  • Reuse existing CSS and layout
  • Each component is independently testable
  • Can stop halfway and still have a working app

Cons

  • Still needs Vite build step for Svelte compilation
  • Two mental models (vanilla + Svelte) coexisting
  • WebSocket store benefits are limited if only some parts use it
  • Ends up as a hybrid — eventually want full migration anyway
C

Vanilla JS + Structured Patterns

Stay framework-free but apply the architectural ideas from the Svelte reference: a WebSocket manager class with reconnection/heartbeat, a reactive state layer, CSS transitions inspired by animations.md. No build step.

Pros

  • Zero build tooling — stays as static files served by FastAPI
  • No compilation, no node_modules, no dev server
  • Simplest deployment — just copy files
  • Smallest diff from current code

Cons

  • Manual DOM updates — no reactivity, no component model
  • Animations stay hand-rolled CSS
  • Harder to maintain as features grow
  • Doesn't leverage the fullstack-guy's Svelte reference patterns
